API Usage

Send your API key in the X-API-Key header:

curl -H "X-API-Key: esa_xxxxxxxx.your_secret_here" \
     https://backend.extraspace.com.sg/api/discount-plans
Response headers: Each API response includes rate limit and quota information:
  • X-RateLimit-Limit — your per-minute rate limit
  • X-Quota-Limit — your daily quota
  • X-Quota-Remaining — remaining requests for today

API Reference — showing endpoints for your assigned scopes

How to choose the right endpoint group
  • 🤖 Chatbot Booking Flow — purpose-built for the chatbot & recommendation engine. Pricing is computed by the engine, idempotent move-ins, automatic SOAP orchestration for perpetual+prepay plans, and session/customer reconciliation back to the recommendation that produced the booking.
  • 📦 SiteLink Reservations (Direct) — thin passthrough to SiteLink for CRUD on existing reservations (list, update, cancel, notes, fee config, etc.) Use when you're not going through the recommender — e.g. ops tooling, back-office integrations, manual reservations.
  • Inventory / Discount Plans / Smart Lock / Statistics / Reservation Fees — supporting data sources. Read-mostly references that any caller can use.
Chatbot booking — 4 calls in order:
  1. POST /api/recommendations — quote engine; read pricing.total_due_at_movein + customer_disclosure.fine_print.
  2. POST /api/reservations/reserve — hold the unit; pass session_id + customer_id for outcome reconciliation.
  3. GET /api/reservations/move-in/cost — SOAP-truth confirmation right before charging.
  4. POST /api/reservations/move-in — pass the total as payment_amount. Send Idempotency-Key: <UUID> to make retries safe (24h cache; replays return idempotent_replay: true).
Perpetual+prepay plans: the middleware fans out the follow-up SOAP calls internally (PaymentSimpleCash for the prepay surplus, ScheduleTenantRateChange for the future ECRI). Bot's contract stays single-amount. See chatbot_integration.md for the full lifecycle.
🔑 Identity & Run Tracking — required on every recommend call

The recommender is stateful per conversation. Four context IDs glue every step together — from the first quote, through follow-up turns, to reserve and move-in. They are how the engine identifies which run / turn it is processing and how outcomes get reconciled back to the recommendation that produced the booking. All four fields are now mandatory on POST /api/recommendations; missing any one returns 400 ValidationError.

Field Cardinality Who mints Purpose
request_idUnique per turnBot — fresh UUID per callIdentifies THIS recommend call. Other turns reference it via previous_request_id. ≤ 64 chars.
session_idStable per conversationBot — once per chatGroups every turn of one chat AND the eventual reserve + move-in. Reconciliation key in mw_recommendations_served. ≤ 64 chars.
customer_idStable per customerChannel-side (e.g. PandaAI user id)Cross-conversation identity. Outcome attribution + per-customer ranking. Use a stable surrogate for anonymous flows. ≤ 64 chars.
channelPer callBotOne of chatbot / web / api / admin. Plans whose distribution_channel excludes this value are dropped.
previous_request_idOptionalBot — set on follow-up turns onlyThis is how the engine knows "this is a continuation". Server fetches the prior turn's filters as the BASE, layers current filters on top, and excludes already-served unit_ids. Pair with picked_slot + action to drive the relax strategy.
How the engine identifies which run / turn it is in
  • Turn 1 — previous_request_id absent → fresh pool fetch, strict filters, relax_strategy_used = "none".
  • Turn 2+ — previous_request_id set → resume_session() fetches the prior row from mw_recommendations_served, inherits its filters, layers current overrides, excludes prior unit_ids. relax_strategy_used = size_plus_one / expand_unit_type / next_nearest_site / wider_size_band based on picked_slot + action.
  • Every recommend call writes one row to mw_recommendations_served. Parent-child chain is previous_request_id → request_id. Conversation grouping is session_id. Booking attribution is session_id + customer_id on /reserve + /move-in.
Worked example — 3-turn conversation ending in a booking 
// Turn 1 — fresh recommend
POST /api/recommendations
{
  "duration_months": 6,
  "filters": { "location": ["L017"], "size_range": ["30-35"] },
  "context": {
    "channel":     "chatbot",
    "request_id":  "req-aaaa-1111",   // fresh UUID
    "session_id":  "sess-conv-7",     // bot mints once per chat
    "customer_id": "pandai_xyz_123"   // stable per customer
  }
}
// → response.next_turn = { session_id, request_id: "req-aaaa-1111", ... }
// → response.stats.relax_strategy_used = "none"

// Turn 2 — customer wants more like slot 1
POST /api/recommendations
{
  "duration_months": 6,
  "filters": { "location": ["L017"], "size_range": ["30-35"] },
  "context": {
    "channel":             "chatbot",
    "request_id":          "req-bbbb-2222",   // NEW per turn
    "session_id":          "sess-conv-7",     // SAME as turn 1
    "customer_id":         "pandai_xyz_123",  // SAME
    "previous_request_id": "req-aaaa-1111",   // ← the link
    "picked_slot":         1,
    "action":              "more_like_this"
  }
}
// → server inherits turn 1 filters, excludes turn 1's unit_ids,
//   relax_strategy_used = "size_plus_one"

// Turn 3 — customer accepts a slot, reserve + move-in
POST /api/reservations/reserve
{
  "site_code": "L017", "unit_id": 107197, "concession_id": 11872,
  "first_name": "Jane", "last_name": "Tan", "phone": "+65...",
  "needed_date": "2026-05-01",
  "session_id":  "sess-conv-7",     // ← link booking → recommendation
  "customer_id": "pandai_xyz_123",  // ← same
  "previous_request_id": "req-bbbb-2222"  // turn that produced the pick
}
// → tenant_id, waiting_id
